# jgfx

A convenience wrapper around **WebGPU** for the browser — a JavaScript port of
[`cgfx`](../../C/webgpucppv2/cgfx). Same mental model, idiomatic JS classes, no build
step.

jgfx handles the WebGPU boilerplate (device setup, frame lifecycle, buffer/shader
creation) while staying a *thin* layer: every wrapper exposes its raw `GPU*` handle, so
you can drop to plain WebGPU whenever jgfx doesn't cover your case.

> **Status:** feature-complete port. The core render path, buffers/uniforms, mesh/math/
> camera, textures/samplers, compute/MRT, and the mesh loader are all implemented, each
> with a runnable example. See [`PLAN.md`](PLAN.md) for the full design and roadmap, and
> the [documentation](#documentation) for guides and the API reference.

## Requirements

- A browser with **WebGPU** (a recent Chromium-based browser, or any browser with
  WebGPU enabled).
- The library is **plain ESM** — no bundler, no build. Because WebGPU needs a secure
  context and examples use `fetch`, you must serve over `http://localhost` (not
  `file://`).

## Running the examples

From the repository root:

```sh
python3 -m http.server      # or: npx serve
```

Then open <http://localhost:8000/examples/> and pick an example.

## Using the library

```js
import { Context } from "./jgfx/index.js";

const canvas = document.querySelector("canvas");
const ctx = await Context.create({ canvas });

const shader = ctx.createShader("tri", wgsl);
const pipeline = ctx.createPipeline({ shader });

function render() {
  const frame = ctx.beginFrame([0.1, 0.1, 0.2, 1]);
  if (frame) {
    frame.pass.setPipeline(pipeline);
    frame.pass.draw(3);
    ctx.endFrame(frame);
  }
  requestAnimationFrame(render);
}
requestAnimationFrame(render);
```

## Documentation

Full docs live in [`docs/`](docs/) — a home page, a getting-started walkthrough, an
architecture overview, four guides, and a per-module API reference. They render as plain
Markdown on GitHub, and build into a searchable themed site (the same
[Material for MkDocs](https://squidfunk.github.io/mkdocs-material/) theme cgfx uses):

```sh
pip install -r requirements-docs.txt
mkdocs serve      # live preview at http://127.0.0.1:8000
```

Documentation tooling is optional — it is never needed to use the library. Start at
[`docs/index.md`](docs/index.md).

## Single-file build (optional)

The source is the primary artifact — import `jgfx/index.js` directly and hack on it
with **nothing installed**. For vendoring jgfx as one file, an optional zero-dependency
amalgamator stitches the modules together (in the spirit of SQLite's `sqlite3.c`):

```sh
node build.mjs      # or: npm run build
```

It writes three files to `dist/` (all committed, all generated — do not hand-edit):

| File                 | Use                                                              |
| -------------------- | --------------------------------------------------------------- |
| `dist/jgfx.js`       | Readable ESM bundle — `import { Context } from "./dist/jgfx.js"` |
| `dist/jgfx.min.js`   | Same, lightly minified (comments/blank lines stripped)          |
| `dist/jgfx.global.js`| IIFE exposing `window.Jgfx` for a plain `<script src>` (no modules) |

```html
<!-- ESM -->
<script type="module">
  import { Context } from "./dist/jgfx.js";
</script>

<!-- or a classic global -->
<script src="./dist/jgfx.global.js"></script>
<script>const ctx = await Jgfx.Context.create({ canvas });</script>
```

Load `dist/smoke.html` in the browser to verify a fresh build (checks both bundles and
draws a triangle). The build has **no dependencies** — plain Node reads `jgfx/*.js`.

## Layout

```
jgfx/        the library (ESM modules, one file per concept)
examples/    one folder per example; open examples/index.html for the menu
dist/        optional single-file bundles (generated by build.mjs)
build.mjs    zero-dependency amalgamator (source → dist/)
docs/        ported prose docs (added in Phase 6)
PLAN.md      full design + phased roadmap
```

## Relationship to cgfx

jgfx keeps cgfx's module boundaries and philosophy (transparency, zero-init defaults, no
global state, caller-owned loop, explicit `destroy()`). Differences forced by the
platform — async context creation, async buffer reads, `requestAnimationFrame` instead
of a native loop, canvas instead of GLFW — are documented in [`PLAN.md`](PLAN.md) §2.
